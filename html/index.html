How this site runs

Node, cluster, ingress, registry, DNS and image build are managed with Terraform and Github Actions in a private repo. The site itself is a static HTML/CSS page, built with buildx and pushed to a self-hosted Harbor registry. The Deployment spec is managed by Terraform, and the GitHub Actions workflow builds the image, updates the image tag, and restarts the Deployment.

Push to merge, live in ~44 seconds: build, registry push, rollout restart.

AI wrote the HTML/CSS on this page. I do infra, not frontend.

Cluster

One Talos node. Control plane and worker, same box (allowSchedulingOnControlPlanes). No managed Kubernetes underneath. Terraform's talos provider bootstraps it and outputs the kubeconfig as state.

Ingress & TLS

ingress-nginx as a DaemonSet, hostNetwork: true, bound to the node's public IP directly. No load balancer. Each domain gets its own cert-manager ClusterIssuer, HTTP-01 against Let's Encrypt.

Build & deploy

Push to site source triggers GitHub Actions: build with buildx, push to a self-hosted Harbor registry, then kubectl rollout restart using a kubeconfig from a repo secret. Terraform owns the Deployment spec. The workflow only changes the image tag.

DNS

vin.moe's A record points at the node through the Cloudflare provider, grey-clouded. Let's Encrypt's HTTP-01 solver needs to reach the origin directly — orange-clouded would block it.
