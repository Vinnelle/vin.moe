How this site runs

The node, the cluster, ingress, the container registry, DNS, the image build, and the Deployment spec are all Terraform, applied through HCP Terraform, in a private repo. The site itself is a static HTML/CSS page.

Push to merge, live in ~44 seconds: build, registry push, rollout restart.

AI wrote the HTML/CSS on this page. I do infra, not frontend.

Cluster

One Talos node. Control plane and worker, same box (allowSchedulingOnControlPlanes). No managed Kubernetes underneath. The node itself is provisioned out-of-band; Terraform bootstraps it with the siderolabs/talos provider and stores the resulting kubeconfig as state.

Registry

Self hosted Harbor instance on the same cluster. This site gets its own harbor_project. Pushes and pulls go through a CI robot account. The robot's password is a Terraform random_password, never typed anywhere by hand, packaged into a dockerconfigjson secret in the websites namespace and mounted as an imagePullSecret.

Image

Two-stage Dockerfile. Stage one is Alpine plus imagemagick, converting the source PNG into a multi-resolution favicon.ico (16/32/48/64). Stage two is nginx:alpine with nginx.conf, the static HTML, and the favicon from the first stage copied in. The nginx config is one server block: try_files with a 404 fallback, no other routes.

Ingress & TLS

ingress-nginx runs as a DaemonSet with hostNetwork: true, bound directly to the node's public IP. No load balancer — one node.

Each domain has its own cert-manager ClusterIssuer, solving via DNS-01 against Cloudflare (same API token, stored as a Kubernetes secret), not HTTP-01. The DNS record for this domain is proxied through Cloudflare rather than pointed at the origin directly, and DNS-01 doesn't require the origin to be reachable — it only needs to write a TXT record. The zone's SSL mode is set to strict, so Cloudflare only forwards traffic to the origin if it presents a currently-valid cert, not a self-signed one and not Cloudflare's own origin CA. TLS is independently valid on both hops: browser to Cloudflare edge, Cloudflare edge to origin.

Deployment & resources

Two replicas of nginx:alpine behind a ClusterIP Service, 50m CPU / 16Mi memory requested, 200m / 64Mi as the limit, readiness probe hitting / every 10 seconds. A VerticalPodAutoscaler in Auto mode adjusts the memory request within that 16–64Mi band based on observed usage. CPU is not managed by the VPA.

Build & deploy

A push to the site's path on prd (or a manual workflow_dispatch) triggers GitHub Actions: log into Harbor with the CI robot's credentials, buildx build and push back to the same :latest tag. The workflow then writes a kubeconfig from a repo secret and runs kubectl rollout restart followed by rollout status --timeout=120s. The tag never changes, so the restart is what forces Kubernetes to re-pull the image.

Mirror

A second job in the same workflow rsyncs two things out to a separate public GitHub repo, using a scoped token: the site source into site-image/, and the workflow file itself into github-workflow/. It commits and pushes only if something actually changed. So the source lives in two places: the private infra monorepo, where it's actually written and built, and a public read-only mirror for anyone who wants to see what's rendering this page.

DNS

vin.moe's A record points at the node's IP through a proxied Cloudflare record. Combined with DNS-01 above, none of Let's Encrypt's validation traffic ever needs to reach the origin at all; the only thing that has to be reachable is the origin's own valid cert, and only Cloudflare needs to see it.
